# Техническое задание и архитектурная спецификация на разработку бэкенда для системы управления глэмпингом

## 1. Архитектурное видение и технологический фундамент

Разрабатываемая система представляет собой высоконагруженный программно-аппаратный комплекс, предназначенный для полной автоматизации взаимодействия между гостями индивидуальных домиков (глэмпов) и обслуживающим персоналом. Инфраструктура требует строгой синхронизации между физическим миром (статус домика, шлагбаумы, доставка физических объектов) и цифровыми интерфейсами (гостевые планшеты в режиме Kiosk Mode, административные панели).

В качестве основы технологического стека утверждена экосистема **Node.js** с использованием фреймворка **NestJS**. Выбор NestJS продиктован его модульной архитектурой, строгой поддержкой TypeScript и встроенными механизмами внедрения зависимостей (Dependency Injection), что предоставляет корпоративный уровень масштабируемости, предотвращая деградацию кодовой базы при добавлении новых интеграций (например, внешних клининговых служб). Реляционная база данных **PostgreSQL** выбрана за надёжность транзакций и поддержку бинарного формата JSONB, который критически важен для хранения динамических схем услуг и прав доступа.

Связующим звеном между базой данных и бизнес-логикой выступает **Prisma ORM**. Prisma генерирует типобезопасный клиент, исключающий ошибки несоответствия типов на этапе компиляции. Конфигурация Prisma (модуль `prisma-client-js`) интегрирована с форматом CommonJS, что обеспечивает максимальную стабильность в связке с экосистемой тестирования Jest без необходимости настройки экспериментальных флагов ESM-модулей. Для инфраструктуры развёртывания используется **Docker**, инкапсулирующий PostgreSQL и, опционально, интерфейс pgAdmin в изолированные контейнеры, что унифицирует среды разработки и продакшена.

Анализ архитектурных паттернов показывает, что система ориентирована на событийно-управляемую логику (Event-Driven Architecture) в области клиентских взаимодействий (чаты, экстренные вызовы) при сохранении классического REST-интерфейса для операций CRUD. Защита качества кодовой базы реализована через многоуровневый пайплайн: ESLint проверяет статический код, Prettier обеспечивает единообразие форматирования, а Husky в связке с lint-staged блокирует отправку дефектного кода в репозиторий. Сгенерированный код Prisma исключён из линтинга (`.eslintignore`), что предотвращает ложные срабатывания хуков на pre-commit стадии.

## 2. Доменная модель и структура базы данных

Структура данных спроектирована с учётом строгой нормализации для транзакционных сущностей и денормализации (через поля JSON) для конфигурационных данных. Это предотвращает избыточные операции JOIN и обеспечивает высокую производительность чтения.

### 2.1. Узел проживания: физические объекты и сессии

Базовым паттерном системы является отказ от прямой привязки пользователей к домикам. Вместо этого вводится промежуточная сущность «Сессия», что позволяет сохранять исторические данные без нарушения приватности текущих гостей.

| Сущность (таблица) | Описание | Механизмы и зависимости |
|---|---|---|
| `houses` | Физические строения (тенты, домики). | Содержит уникальный идентификатор `number` и аппаратный токен `device_token` для авторизации привязанного к стене планшета. Связана с сессиями отношением 1:N. |
| `guest_sessions` | Исторический и текущий контекст проживания. | Инкапсулирует период заезда (`check_in_date`) и выезда (`check_out_date`). Маркер `is_active` оптимизирует поиск. |

Наличие поля `is_active` в таблице `guest_sessions` сопровождается композитным B-Tree индексом `@@index([house_id, is_active])`. Данное архитектурное решение генерирует существенный системный эффект: при каждом запросе от любого домика серверу необходимо верифицировать текущую сессию. Индекс позволяет PostgreSQL осуществлять поиск за логарифмическое время, игнорируя тысячи закрытых архивных сессий.

### 2.2. Узел управления персоналом: модель RBAC

Управление доступом персонала строится на принципах Role-Based Access Control (RBAC).

| Сущность (таблица) | Описание | Механизмы и зависимости |
|---|---|---|
| `roles` | Ролевые профили (Администратор, Клининг). | Использует поле `permissions` типа JSON для хранения массива строковых ключей доступа (например, `["manage_users", "view_tasks"]`). Подобная структура позволяет вводить новые права без SQL-миграций. |
| `users` | Учётные записи штатного персонала. | Пароли хранятся в поле `password_hash` (алгоритм Argon2 или Bcrypt). Поле `refresh_token` хранит JWT-токен для длительных сессий. Связь с `roles` через `role_id`. |

Первичное наполнение базы данных администратором исключено из публичного API для устранения уязвимости перехвата контроля (First User Takeover). Создание суперпользователя делегировано Prisma Seed-скриптам, использующим операции `upsert`.

### 2.3. Узел обслуживания: каталог услуг и динамические заявки

Инфраструктура глэмпинга требует обработки широкого спектра задач (от доставки дров до вызова такси), атрибуты которых кардинально различаются.

| Сущность (таблица) | Описание | Механизмы и зависимости |
|---|---|---|
| `services` | Номенклатура доступных сервисов. | Содержит логические флаги `requires_time` и ценовую информацию `price_info`. Поле `json_schema` хранит стандартизированную схему валидации для поля `details` заявки. |
| `tasks` | Операционная заявка (тикет Канбан-доски). | Центральная транзакционная сущность. Отслеживает статус (`TaskStatus`), привязку к дому (`house_id`) и сессии (`session_id`). Динамические требования клиента сохраняются в `details` (JSON). |

Связывание `tasks` с `guest_sessions` обеспечивает автоматическую изоляцию данных. Каскадное удаление или архивация сессии позволяет мгновенно блокировать заявки уехавших гостей. Поле `price_fix` выступает иммутабельным финансовым слепком, фиксирующим стоимость услуги в момент её заказа, что защищает аналитику при будущих изменениях прайс-листа.

### 2.4. Узел ресторана: управление меню и заказами

Логика работы ресторана частично интегрирована в общую систему задач, но обладает собственной детализированной структурой для управления номенклатурой.

| Сущность (таблица) | Описание | Механизмы и зависимости |
|---|---|---|
| `meal_types` | Окна обслуживания (Завтрак, Обед). | Ограничивает доступность меню через `start_time` и `end_time` (`@db.Time(6)`). Связана с меню через неявное соединение M:N. |
| `menu_items` | Блюда и напитки. | Цена `price` хранится в минимальных единицах (integer), исключая ошибки округления стандарта IEEE 754. Флаг `is_available` позволяет моментально скрывать позиции. |
| `food_order_items` | Детализация позиций внутри заказа. | Связана с таблицей `tasks` (где задача выступает чеком) через `task_id` с правилом `onDelete: Cascade`. Фиксирует `quantity` и `price_at_order`. |

Архитектура предполагает, что заказ еды генерирует единый агрегированный `task`, внутри которого вложены множественные `food_order_items`. Это позволяет администраторам отслеживать статус приготовления еды в едином потоке Канбан-доски наряду с уборкой и трансфером.

### 2.5. Узел коммуникаций и настроек

Связь между гостями и рецепцией требует журналирования и мгновенной доставки.

| Сущность (таблица) | Описание | Механизмы и зависимости |
|---|---|---|
| `chat_messages` | История переписки. | Поле `sender_type` дифференцирует автора (`GUEST` / `STAFF`). Жёсткая связь с `session_id` (`onDelete: Cascade`) гарантирует уничтожение данных при удалении сессии, обеспечивая приватность. |
| `settings` | Глобальные переменные. | Ключ-значение (`key`, `value`) для хранения паролей Wi-Fi, инструкций и текстовых приветствий, загружаемых на планшет при инициализации. |

## 3. Матрица аутентификации и авторизации

Безопасность системы разделена на два непроницаемых контура: stateless-аутентификацию для мобильных устройств персонала и token-based идентификацию для стационарных планшетов.

### 3.1. Контур персонала (JWT & Passport)

Персонал взаимодействует с системой через административную веб-панель. Аутентификация построена на механизмах Passport.js и стандартах JSON Web Token (JWT).

После проверки пары логин/пароль с помощью хэширования (`argon2.verify`) сервер генерирует короткоживущий Access Token и долгоживущий Refresh Token. Секретные ключи (`JWT_SECRET`) и параметры времени жизни подгружаются строго из среды выполнения через защищённый `ConfigModule`. Токен обновления сохраняется в базе данных для реализации механики ротации токенов (Refresh Token Rotation), предотвращающей бессрочный доступ при краже ключа.

Маршруты API персонала защищаются глобальными или локальными гвардами (Guards):

- **`JwtAuthGuard`** — наследует `AuthGuard('jwt')`. Перехватывает запрос, валидирует подпись токена, извлекает идентификатор пользователя и интегрирует объект пользователя в `ExecutionContext` (`request.user`).
- **`PermissionsGuard`** — реализует ролевую модель (RBAC). При помощи декоратора `@RequirePermissions('manage_users')` метаданные привязываются к маршруту. Guard использует класс `Reflector` для извлечения метаданных и сопоставляет их с массивом `request.user.role.permissions`. При отсутствии пересечений генерируется `ForbiddenException`.

### 3.2. Контур гостевых терминалов (Device Token)

Планшеты работают автономно в режиме киоска (Kiosk Mode). Интерфейс ввода учётных данных отсутствует, что диктует необходимость аппаратной авторизации.

В базу данных при инсталляции планшета зашивается неизменяемый `device_token`. Каждый HTTP-запрос от киоска сопровождается передачей этого токена в специализированном заголовке. Архитектура предусматривает класс `DeviceGuard`:

1. Извлечение токена из заголовка.
2. Обращение к `PrismaService` для поиска записи `houses` с соответствующим `device_token` и загрузка вложенной связи `guest_sessions`, где `is_active` равно `true`.
3. Если активная сессия найдена, объекты `house` и `session` прикрепляются к объекту `Request`, открывая контроллерам доступ к идентификатору сессии.

Это устраняет необходимость передачи `session_id` со стороны недоверенного клиента (планшета), закрывая уязвимость подмены сессий (Insecure Direct Object Reference).

## 4. Парадигма динамической валидации (AJV и JSON Schema)

Одной из самых технологически сложных задач является обеспечение гибкости каталога услуг. Жёстко типизированные DTO (Data Transfer Objects) не позволяют администраторам «на лету» создавать услуги со специфическими полями (например, запрос типа детского кресла для трансфера или степени прожарки стейка).

Решение базируется на спецификации JSON Schema и сверхбыстром компиляторе **AJV** (Another JSON Validator).

### 4.1. Механизм работы AJV

В сущности `services` предусмотрено поле `json_schema`, хранящее стандартизированное описание ожидаемых данных. AJV преобразует эти декларативные схемы в высокооптимизированные императивные JavaScript-функции валидации, что обеспечивает задержку на уровне микросекунд даже при огромных нагрузках.

Поскольку компиляция схемы (`ajv.compile()`) требует вычислительных ресурсов, архитектура предписывает паттерн кэширования. Скомпилированные валидаторы сохраняются в памяти сервера (через механизмы инстанцирования AJV).

### 4.2. Интеграция в конвейер NestJS

Для бесшовной интеграции разрабатывается кастомный `PipeTransform`:

1. `DynamicSchemaValidationPipe` перехватывает тело `POST /tasks`.
2. Извлекает `service_id` и запрашивает соответствующую `json_schema` из кэша или БД.
3. Запускает скомпилированную функцию валидации. При успехе запрос пропускается к контроллеру.
4. При несоответствии данных (например, отсутствие обязательного поля или нарушение формата даты) AJV генерирует массив ошибок (`validate.errors`), который Pipe трансформирует в стандартизированное исключение `BadRequestException`, описывающее точную причину для отображения на фронтенде.

Этот системный эффект позволяет фронтенду (использующему библиотеки вроде `react-jsonschema-form`) динамически генерировать графический интерфейс (поля ввода, чекбоксы) исключительно на базе JSON Schema, превращая бэкенд в полноценный Headless-движок услуг.

## 5. Декомпозиция бизнес-логики и маршрутизации

Бизнес-логика разделена на изолированные домены (модули). Каждый модуль инкапсулирует контроллеры (для обработки маршрутов), сервисы (для логики и ORM-взаимодействия) и DTO.

### 5.1. Модуль управления проживанием (Check-in / Check-out)

Жизненный цикл сессии формирует контекст для всех остальных операций.

- **Инициализация (Check-in):** при POST-запросе от администратора система обращается к `PrismaService` и выполняет проверку на наличие активных сессий в указанном домике. При отсутствии таковых создаётся новая запись `guest_sessions`.
- **Терминация (Check-out):** сложнейший транзакционный узел. Сервис переводит флаг `is_active` в `false`. В рамках этой же SQL-транзакции система осуществляет поиск всех незавершённых задач (`tasks` в статусе `NEW` или `IN_PROGRESS`) для данной `session_id` и принудительно переводит их в статус `CANCELLED`. Это гарантирует отсутствие «висячих» процессов на Канбан-доске после отъезда гостей.

### 5.2. Модуль задач и Канбан-процессор

Ядром диспетчеризации выступает сущность Task.

Маршруты поддерживают глубокую фильтрацию через параметры запроса (Query Params). Администраторский `GET`-запрос может включать фильтры по статусу, исполнителю или типу услуги.

Архитектура предполагает изоляцию данных на уровне базы данных: если к API обращается пользователь с ролью «Клининг», слой сервиса модифицирует объект `where` в запросе Prisma, добавляя принудительное условие на `service_id`, соответствующее услугам уборки. Такой подход устраняет риск утечки данных, так как фильтрация происходит в ядре PostgreSQL, а не при итерации массива в памяти Node.js. Любое изменение статуса задачи (`PATCH /tasks/:id/status`) триггерит запись в журнал аудита.

### 5.3. Модуль ресторанного обслуживания

Логика питания требует валидации в двух пространственных измерениях: времени и финансах.

- **Валидация времени:** кастомный Pipe или сервисная логика проверяет `requested_time` из DTO. Значение должно быть удалено от `Date.now()` минимум на 1 час (требование кухни). Дополнительно `requested_time` сопоставляется с интервалами, загруженными из связанной `meal_types` (например, 08:00–10:00 для завтрака).
- **Заморозка транзакции:** при оформлении заказа сервис перебирает массив `menu_item_id`. Для каждого элемента извлекается актуальный `price` из `menu_items`. Этот прайс жёстко записывается в `price_at_order` таблицы `food_order_items`. Это гарантирует абсолютную иммутабельность финансовой истории чека.

### 5.4. Модуль интеграции гео-трансфера

Модуль трансфера демонстрирует принцип недоверия к клиентским устройствам. Гость на планшете вводит только целевой адрес. Все вычисления производятся на бэкенде:

1. `PricingService` отправляет гео-координаты во внешнее API (Яндекс.Маршрутизация / 2GIS).
2. На основе полученной длины маршрута применяется тарифная сетка (до 35 км, до 50 км, свыше 50 км).
3. Итоговая стоимость фиксируется в поле `price_fix` создаваемой задачи.

### 5.5. Модуль коммуникации (WebSockets/SSE)

Для обеспечения мгновенной реакции на экстренные вызовы (кнопка «Открыть ворота») и сообщения в чате применяется архитектура двунаправленного соединения. Через `WebSocketGateway` в NestJS администраторы подписываются на системные комнаты. При создании записи `chat_messages` сервис не только выполняет операцию сохранения в БД (с каскадной привязкой к сессии), но и отправляет broadcast-событие на сокет-клиенты администраторов, обеспечивая UI-реактивность.

## 6. Глобальные слои перехвата и обработки (Interceptors & Pipes)

Для соответствия корпоративным стандартам разработана система глобальных перехватчиков, гарантирующая, что API всегда возвращает унифицированную структуру JSON независимо от характера ответа.

### 6.1. Глобальная валидация (ValidationPipe)

Модуль `ValidationPipe` интегрирован на уровне приложения (`app.useGlobalPipes`). Используется пакет `class-validator`.

Ключевые конфигурации:

- **`whitelist: true`** — отбрасывает любые поля JSON-запроса, которые явно не определены в DTO, ликвидируя риск внедрения скрытых свойств (Mass Assignment).
- **`transform: true`** — автоматически приводит входящие строковые данные (например, `id` в URL) к требуемым типам (числа, объекты дат), исключая необходимость ручного парсинга в контроллерах.

### 6.2. Обработка исключений (AllExceptionsFilter / HttpExceptionFilter)

Разработан класс, перехватывающий все исходящие ошибки. Если выброшено стандартное исключение (`HttpException`), фильтр транслирует его статус и сообщение. В случае системной ошибки (Internal Server Error / 500) фильтр скрывает стек-трейс от конечного пользователя (во избежание утечки архитектурных деталей), но направляет полную информацию в логгер на сервере. Формат ответа строго структурирован:

```json
{
  "success": false,
  "statusCode": 500,
  "timestamp": "...",
  "path": "...",
  "error": "..."
}
```

### 6.3. Модификация ответов (ResponseInterceptor)

`ResponseInterceptor` использует TypeScript Generics (`NestInterceptor<T, ApiResponse<T>>`) для обёртывания всех успешных ответов. Интерцептор перехватывает результат выполнения сервиса (поток Observable) и мапит его в структуру:

```json
{
  "success": true,
  "data": "...",
  "timestamp": "..."
}
```

Применение оператора нулевого слияния (`??`) вместо логического ИЛИ (`||`) критически важно для сохранения корректных возвращаемых значений, таких как `0` или `false`.

### 6.4. Журналирование и мониторинг (LoggingInterceptor)

Логирование каждого HTTP-запроса реализуется на уровне перехватчика. Фиксируются IP-адрес источника, метод маршрута, путь и время выполнения в миллисекундах (`Date.now()`). Использование оператора `catchError` в цепи RxJS гарантирует фиксацию в логах не только успешных, но и аварийно завершённых транзакций.

## 7. Конфигурация и безопасность (Environment & Joi)

Запуск системы блокируется на самом раннем этапе при отсутствии критических настроек (паттерн Fail-Fast). В `app.module.ts` интегрирован `ConfigModule`. Валидация структуры файла `.env` доверена библиотеке Joi.

Если в среде выполнения отсутствуют переменные `DATABASE_URL`, `JWT_SECRET` или параметры времени жизни токенов, Joi генерирует фатальное исключение `Config validation error` до момента инициализации пула подключений к БД. Файл `.env` исключён из системы контроля версий, для синхронизации структуры между разработчиками используется безопасный `.env.example`.

Безопасность сетевого уровня обеспечивается интеграцией модуля **Helmet** (для предотвращения атак типа XSS и сниффинга MIME-типов посредством установки защищённых HTTP-заголовков) и настройкой механизма **CORS** (`app.enableCors`) с поддержкой `credentials: true` для кросс-доменного обмена сессионными куками и заголовками.

Документация API генерируется автоматически через `@nestjs/swagger`. DTO аннотируются декораторами `@ApiProperty()`, формируя интерактивный графический интерфейс по адресу `/api/docs`, что кардинально упрощает интеграцию с фронтенд-командами.

## 8. Поэтапный план реализации (Roadmap)

Для обеспечения непрерывной интеграции и тестирования архитектура должна внедряться поэтапно.

### Подготовительный этап: Инфраструктура

- Инициализация проекта: настройка NestJS, установка линтеров (ESLint, Prettier, Husky, lint-staged). Добавление сгенерированных файлов Prisma в `.eslintignore`.
- Конфигурация БД: развёртывание `docker-compose.yml` (PostgreSQL), инициализация Prisma, базовая миграция.
- Глобальные механизмы: настройка Joi, внедрение Interceptors, Filters, Pipes и Swagger.

### Фаза 1: Ядро и безопасность

- `AuthModule`: реализация `JwtStrategy` для персонала и `DeviceGuard` для киосков. Создание Prisma Seed для первоначального администратора.
- Проживание: эндпоинты `/check-in` и `/check-out` с механизмом автоматического терминирования сессий и изоляции истории.

### Фаза 2: Операционная логика

- Канбан и Tasks: CRUD для заявок, фильтрация через Prisma, ролевая изоляция на уровне базы данных.
- Ресторанный модуль: загрузка меню, логика корзины (`food_order_items`), валидация временных слотов, фиксация цен `price_at_order`.
- Внешние интеграции: модуль калькуляции трансфера через Yandex API; настройка WebSockets для чата и ворот.

### Фаза 3: Масштабирование и Production

- Динамические услуги: подключение AJV. Внедрение JSON Schema в `services` и создание `DynamicSchemaValidationPipe`.
- Файловая система: настройка S3 (или MinIO в Docker) и библиотеки Multer для управления изображениями блюд.
- Развёртывание: интеграция Dockerfile для Production, настройка Graceful Shutdown для безопасного закрытия Prisma-пулов, разработка Unit/E2E тестов.

## 9. Системные эффекты и резюме масштабируемости

Архитектурный фундамент, заложенный в данном ТЗ, исключает накопление технического долга благодаря нескольким ключевым факторам:

- **Неизменность прошлых состояний:** механизмы `price_at_order` и `price_fix` позволяют изменять прайс-листы и бизнес-правила (схемы `json_schema`) в настоящем времени без разрушения ретроспективной аналитики за прошлые периоды.
- **Делегирование UI-логики:** перенос структуры услуг в `json_schema` трансформирует бэкенд в Headless-систему, позволяя администраторам генерировать интерфейсы для планшетов динамически, исключая необходимость релиза мобильных приложений.
- **Гранулярность доступа:** реализация RBAC через массивы JSON вместо жёсткой реляционной структуры позволяет масштабировать ролевую модель глэмпинга на любое количество подрядчиков (кейтеринг, спа-салоны) без миграций базы данных.

Данное техническое задание полностью обеспечивает потребности бизнеса в автоматизации и закладывает прочную инженерную базу, соответствующую стандартам Enterprise-разработки в экосистеме Node.js.
